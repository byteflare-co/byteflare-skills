# Playbook

ブラウザ操作の手順書（Playbook）を管理・実行する統合スキル。

## ファイル配置ルール（重要）

**すべての Playbook データは `.claude/skills/playbook/references/` に配置すること。**

これは、このスキル自身のディレクトリ内の `references/` サブディレクトリにあたる。SKILL.md と同じ場所に生成されるため、プロジェクトの git で一緒に管理・共有できる。

生成するファイル：
- `.claude/skills/playbook/references/ui-map.md` — UI Map + Playbookマップ（`/playbook init` で生成）
- `.claude/skills/playbook/references/[name].md` — 個別Playbook（`/playbook create` で生成）

ディレクトリが存在しない場合は `mkdir -p .claude/skills/playbook/references` で作成すること。

## サブコマンド判定

$ARGUMENTS の内容で分岐する：

| 入力 | アクション |
|------|----------|
| `init [URL]` | → UI Map 生成フロー |
| `create` | → Playbook 作成フロー |
| `list` | → Playbook 一覧表示フロー |
| `resync [画面名]` | → UI Map 再スキャンフロー |
| `update` | → Playbook 更新フロー（個別 Playbook の手動編集用） |
| それ以外 / 省略 | → Playbook 実行フロー |

**`resync` と `update` の違い**:
- **`resync`**: UI の実態を再スキャンして自動的に差分を検出し `ui-map.md` を更新する。UI 変更への追従用。
- **`update`**: 既存 Playbook や ui-map.md を手動で編集するフロー。ドメイン知識の追加・制約の追記など、ブラウザスキャンでは得られない情報の更新用。

**いずれのサブコマンドでも、まず `.claude/skills/playbook/references/ui-map.md` の存在を確認する。**
存在しない場合（`init` 以外）→ 「先に `/playbook init [URL]` を実行してUI Mapを作成してください。」と伝えて終了する。

## init：UI Map 生成フロー（`/playbook init [URL]`）

対象プロダクトのUIを巡回し、`.claude/skills/playbook/references/ui-map.md` を生成する。

### ステップ1：URLの確認
- $ARGUMENTS から URL を取得する
- URLがない場合は「対象プロダクトのURLを教えてください。ログインが必要なら、先にログイン済みの状態にしておいてください。」と聞く

### ステップ2：画面リストの提示
1. URLに navigate し、read_page でアクセシビリティツリーを取得する
2. ナビゲーション要素（サイドバー、ヘッダーメニュー、タブなど）から到達可能な主要画面のリストを提示する
3. 「以下の画面が見つかりました。全部カバーしますか？特定の画面だけでいいですか？」と聞く

### ステップ3：巡回 & キャプチャ
対象画面を1つずつ巡回する。各画面で：
1. ページに navigate する
2. 2秒待って read_page でアクセシビリティツリーを取得
3. スクリーンショットを撮影（computer ツール使用）
4. 以下を記録：画面名、URL、主要UI要素（role, name, テキストラベル）、セクション構成

巡回ルール：
- 各画面の完了後「✅ [画面名] を記録しました。次に進みます。」と報告
- 長いページは上部・中部・下部の3回に分けて read_page
- モーダルやドロップダウンは開いた状態も記録
- エラーや想定外の画面はユーザーに報告

### ステップ4：ui-map.md 生成
`.claude/skills/playbook/references/` ディレクトリを作成し、`.claude/skills/playbook/references/ui-map.md` を以下の構成で生成する。`作成日` と `最終同期日` はどちらも**今日の日付**を設定する（`date +%Y-%m-%d` で取得）:

```markdown
# [プロダクト名] UI Map
作成日: YYYY-MM-DD
最終同期日: YYYY-MM-DD

## Playbook マップ
（初回は空テーブル。create で Playbook を追加するたびにここにエントリが増える）
| キーワード | ファイル | 概要 |
|-----------|---------|------|

## 画面一覧
| # | 画面名 | URL | 概要 |
|---|--------|-----|------|
| 1 | ... | ... | ... |

## 画面遷移マップ
- ダッシュボード
  - → 「○○」クリック → ○○画面
  - → 「△△」クリック → △△画面

## 共通UI要素
（全画面共通のヘッダー、サイドバー、ナビゲーションなど）

## 画面詳細

### 1. [画面名]
**URL:** /path

#### レイアウト構成
- ヘッダー：[内容]
- サイドバー：[内容]
- メインエリア：[内容]

#### UI要素
| 要素 | 種類 | テキスト/ラベル | 操作 |
|------|------|----------------|------|
| ... | button | 「保存」 | クリックでデータ保存 |

#### 備考
- [その画面特有の注意点]
```

### ステップ5：確認
1. 抜けている画面はありますか？
2. 各画面の説明で修正・補足したい点はありますか？
3. 続けてPlaybookを作成しますか？（`/playbook create`）

### 要素記録の原則
- テキストラベル（「保存」「キャンセル」等）を最優先。色・位置での指定は避ける
- read_page の role, name, description を正確に記録する
- このUI Mapは後でブラウザ自動操作のリファレンスになるため、要素を確実に特定できる情報を優先

## create：Playbook 作成フロー（`/playbook create`）

### フェーズ1：ヒアリング
1. 「どんな操作のPlaybookを作りますか？」と聞く

### フェーズ2：記録（並行して実施）

#### (a) UI Map の読み込み + 追加巡回
- `.claude/skills/playbook/references/ui-map.md` を読み込んで画面構成を把握する
- UI Map にない画面が必要な場合のみ、Agent でバックグラウンド巡回して補完する

#### (b) ドメイン知識のインタビュー（メインの作業）
AskUserQuestion で業務シナリオを深掘りする。まとめて選択式で聞くと効率的。
- 「どの画面から始めますか？」
- 「検索するとき、何を入力しますか？」（← UIには書かれていない）
- 「結果を見た後、次に何をしますか？」
- 「どの値を見て成功/失敗を判断しますか？」（← UIには書かれていない）

#### (c) 制約・制限事項の調査
対象プロダクトの**公式ドキュメント**を WebFetch/WebSearch で取得し、以下を調査する：
- **できること**: 部分一致検索、特定フィールドでの絞り込みなど
- **できないこと**: 数値比較、正規表現、範囲検索など
- **データの保持期間**: 自動削除ルールなど

### フェーズ3：Playbook ファイル生成
`.claude/skills/playbook/references/[name].md` を以下のフォーマットで生成する：

```markdown
# [Playbook名]

## 前提条件
- [ログイン状態、開始画面などの前提]

## 実行前の判断ルール
- 公式ドキュメント: [URL]
- 不可能な場合は操作を始めずに代替案を提示する
- ページネーションで全件確認するような力技は絶対にやらない

## 制約・制限事項
- **[機能名]でできること**: [具体的に]
- **[機能名]でできないこと**: [具体的に。なぜできないかの理由も]
- **データ保持期間**: [自動削除ルールなど]
- **代替手段**: [制約に該当する場合の対処法]

## ドメイン知識
- [UIだけでは分からない業務知識]

## 手順

### ステップ1：[操作の概要]
- **操作**: [具体的な操作内容]
- **対象要素**: role=[role], name="[テキストラベル]"
- **完了確認**: [成功判定の方法]
- **失敗時**: [エラー時の対処]

## 出力
[完了後の報告フォーマット]
```

### フェーズ4：Playbook マップ更新 & 確認
1. `.claude/skills/playbook/references/ui-map.md` の「Playbook マップ」テーブルにエントリを追加する
2. 生成内容をユーザーに見せて確認を取る
3. 「✅ `/playbook [操作名]` で実行できるようになりました。」と報告

### 生成ルール
- **対象要素**はテキストラベルと role を必ず含める。色・位置・CSSセレクタでの指定は禁止
- **完了確認**は省略しない
- **失敗時**は最低でも「リトライ」「ユーザーに報告」のどちらかを書く
- 繰り返し操作がある場合はループの開始・終了条件を明記
- 条件分岐がある場合は `#### [条件]の場合` で分岐を明示

## list：Playbook 一覧表示フロー（`/playbook list`）

現在登録されている Playbook を一覧表示する。

### ステップ1：Playbook マップの読み込み
1. `.claude/skills/playbook/references/ui-map.md` を読み込む
2. 「Playbook マップ」テーブルの内容を抽出する

### ステップ2：一覧表示
以下のフォーマットで出力する：

```
📘 登録済み Playbook 一覧（N件）

| # | キーワード | ファイル | 概要 |
|---|----------|---------|------|
| 1 | ... | ... | ... |
...

実行: /playbook [キーワード]
詳細: .claude/skills/playbook/references/[ファイル名] を参照
```

### ステップ3：補足情報
- Playbook が0件の場合 → 「登録済みのPlaybookはありません。`/playbook create` で作成できます。」と伝える
- Playbook マップテーブルが `ui-map.md` に存在しない場合 → 「`ui-map.md` に Playbook マップが見つかりません。`/playbook init` を再実行するか、手動でテーブルを追加してください。」と伝える

### ステップ4：UI Map 鮮度チェック（自動再スキャン通知）

`ui-map.md` の先頭にある `最終同期日: YYYY-MM-DD` を読み取り、今日との差を計算する。

| 経過日数 | 表示 |
|---|---|
| 30日未満 | 何も表示しない |
| 30〜59日 | `ℹ️ ui-map.md の最終同期から N 日経過しています。UIに変化があれば /playbook resync で同期できます。` |
| 60日以上 | `⚠️ ui-map.md の最終同期から N 日経過しています。Playbook が動かない可能性があります。/playbook resync で再スキャンしてください。` |

`最終同期日` フィールド自体が存在しない場合（旧バージョンの ui-map.md）:
- `ℹ️ ui-map.md に最終同期日が記録されていません。/playbook resync を実行すると最新状態に同期できます。` と通知する

## 実行：Playbook 実行フロー（`/playbook [操作名]`）

### ステップ1：該当Playbookの特定
1. `.claude/skills/playbook/references/ui-map.md` の「Playbook マップ」テーブルを読み込む
2. $ARGUMENTS のテキストとキーワード列を照合し、該当するPlaybookファイルを特定する
3. 該当なしの場合 → Playbookマップ一覧を提示して「どれを実行しますか？」と聞く

### ステップ2：Playbookの読み込みと可否判断
1. `.claude/skills/playbook/references/[name].md` を読み込む
2. 「前提条件」を確認する
3. ユーザーの依頼内容が「制約・制限事項」に該当しないか確認する
4. **該当する場合 → 操作を始めずに制約を説明し、代替案を提示する**
5. 不明な場合 → 「公式ドキュメントURL」を WebFetch で確認してから判断する

### ステップ3：操作を実行
1. 手順セクションに従いブラウザ操作を実行する
2. 各ステップの「完了確認」で成功を検証する
3. 失敗時は「失敗時」の指示に従う

### ステップ4：結果を報告
1. 「出力」フォーマットに従って結果を報告する

## resync：UI Map 再スキャンフロー（`/playbook resync [画面名]`）

対象プロダクトの UI を再巡回し、`ui-map.md` に記録された状態と現在の UI の差分を検出して反映する。UI 変更への追従用。

### ステップ1：対象範囲の決定
- 引数なし → `ui-map.md` の「画面一覧」から全画面を対象
- 引数あり → 指定された画面名に部分一致する画面を対象

対象画面の一覧をユーザーに提示し、続行確認を取る:
```
🔄 resync 対象画面（N件）:
  1. ダッシュボード (/dashboard)
  2. 会員一覧 (/members)
  ...

このまま再スキャンしますか？ [y/n]
```

### ステップ2：再巡回 & 差分検出

各対象画面について以下を実行:

1. 画面の URL に navigate
2. 2秒待って `read_page` でアクセシビリティツリーを取得
3. `ui-map.md` の該当画面の「UI要素」テーブルを読み込む
4. 現在の要素リストと比較し、差分を以下のカテゴリに分類:

| 種別 | 判定基準 | アイコン |
|---|---|---|
| 追加 | 現状にあり、ui-map に無い | 🆕 |
| 削除 | ui-map にあり、現状に無い | 🔴 |
| テキスト変更 | role 同じ、name が変わった（近傍要素で類似判定） | 🟡 |
| role 変更 | name 同じ、role が変わった | 🟡 |
| URL 変更 | 画面自体の URL が変わった | 📍 |

**類似判定のヒューリスティック**（テキスト変更 vs 追加+削除の判別）:
- 削除候補と追加候補のペアがある場合、以下を順に見る:
  1. 同じセクション内にある
  2. 近傍要素（前後の要素）が一致
  3. role が同じ
- いずれも満たせば「テキスト変更」として扱う
- 曖昧な場合はステップ3でユーザーに判断を委ねる

### ステップ3：差分サマリの提示と承認

全画面の差分をまとめて表示する:

```
📊 UI 変更検出サマリ

【会員詳細画面】
  🔴 削除: button「保存」
  🆕 追加: button「確定する」
  → 自動マッチング候補: 「保存」→「確定する」（テキスト変更として扱う？ [y/n]）

【会員一覧画面】
  🟡 テキスト変更: link「ユーザー一覧」→「会員一覧」
  🟡 テキスト変更: button「キャンセル」→「戻る」

【ダッシュボード】
  変更なし
```

AskUserQuestion で承認モードを聞く:
- [1] すべて適用
- [2] 個別に選択（各差分ごとに y/n）
- [3] キャンセル

曖昧な自動マッチング候補があれば、個別に確認する。

### ステップ4：ui-map.md への反映

承認された差分を `ui-map.md` に in-place で反映する:

1. **追加要素** → 該当画面の「UI要素」テーブルに行を追加
2. **削除要素** → 該当行を削除
3. **テキスト変更 / role 変更** → 該当行を編集
4. **URL 変更** → 「画面一覧」と該当画面の `**URL:**` を更新
5. **最終同期日** を今日の日付に更新

完了レポート:
```
✅ ui-map.md を更新しました

📝 反映内容:
  - 会員詳細画面: 削除1, 追加1 (テキスト変更として適用)
  - 会員一覧画面: テキスト変更2

最終同期日: YYYY-MM-DD

⚠️ 注意: 変更された要素を参照している既存 Playbook がある可能性があります。
   次回のリリースで影響 Playbook の自動検出機能を提供予定です。
   当面は以下のコマンドで手動確認してください:
   grep -r "「[変更前のテキスト]」" .claude/skills/playbook/references/
```

### resync の制約・注意点

- **ブラウザが対象プロダクトにアクセスできる状態であること**（ログイン済みなど）
- **動的に変わる UI 要素**（時刻表示、ユーザー名など）は都度差分として検出されるため、ui-map.md 記録時に「動的」フラグを付けるのが理想（将来拡張）
- **モーダルやドロップダウン内の要素**は、開いた状態を再現しないと検出できない。init 時に開いた状態を記録していれば、resync でも同じ手順で開く必要がある
- **フェーズ5（影響 Playbook の自動更新）は未実装**。このバージョンでは ui-map.md のみ更新する

## update：Playbook 更新フロー（`/playbook update`）

1. `.claude/skills/playbook/references/ui-map.md` の「Playbook マップ」を読み込んで一覧を提示する
2. ユーザーに更新対象を選んでもらう（`ui-map.md` 自体の更新も選択可能）
3. 該当ファイルを読み込む
4. 「何を更新しますか？」と聞く（ステップ追加、制約追記、ドメイン知識追加、画面追加など）
5. 更新を反映し、ユーザーに確認を取る
6. Playbook マップのキーワードも必要に応じて更新する

**注意**: `update` は手動編集のためのフローなので、`ui-map.md` の `最終同期日` フィールドは**更新しない**。`最終同期日` は UI の実態を検証した日付を表すため、ブラウザで実際にスキャンした `resync` / `init` 実行時のみ更新する。

## 品質チェックリスト（create / update 時）

- [ ] 全ステップに4要素（操作・対象・確認・失敗時）が揃っている
- [ ] 対象要素は全て role + テキストラベルで指定されている
- [ ] 前提条件が明記されている
- [ ] 「実行前の判断ルール」セクションが含まれている（公式ドキュメントURL必須）
- [ ] 「制約・制限事項」セクションが含まれている
- [ ] 「ドメイン知識」セクションが含まれている
- [ ] Playbook マップ（`.claude/skills/playbook/references/ui-map.md`）が更新されている

## アンチパターン

- ページネーションで全件を順番にチェックする手順
- 数値の大小比較をテキスト検索で実現しようとする手順
- ツールの制約を確認せずに「とりあえずやってみる」手順
- 「見つからなかったら次のページ」を繰り返すループ
